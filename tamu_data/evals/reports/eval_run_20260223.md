## What this eval measures (and what it does NOT)

### What it measures
| Signal | How |
|---|---|
| Router function accuracy | `function_actual == function_expected` (from stratum label) |
| Category confidence calibration | ECE across confidence bins |
| `semantic_intent` precision / recall / F1 | Intent F1 metric |
| Citation rate | `[Source N]` pattern in generated response |
| Retrieval chunk count | Post-dedup chunks returned per query |
| End-to-end latency by function | router / retrieval / generation breakdown |

### What it does NOT measure (the gap)
The golden set contains a `reference_answer` field тАФ the raw source chunk that grounded
each question. The pipeline generates a full response for every question. **We never compare
the generated response to the reference answer.** This means:

- We don't know if the generated answer is faithful to the source material
- We don't know if the correct chunk was retrieved (recall@k)
- We don't know if the answer would actually satisfy the student's question

The pipeline responses ARE stored in `response_preview` and `response_length_chars` in the
JSONL тАФ they exist, they're grounded, they cite sources тАФ but they are evaluated by zero
automated quality metrics beyond "did it produce a citation?"

Example of the gap:
```
Q: What are the prerequisites for CSCE 413?
REFERENCE: "Grade of C or better in CSCE 315 or CSCE 331 or approval of the instructor."
GENERATED: "The prerequisites for CSCE 413, section 500 are a grade of C or better in
            CSCE 315 or CSCE 331, or approval of the instructor [Source 1]."
```
The generated answer is correct and grounded. But we score this test case as FAIL because
the router chose `metadata_specific` instead of the golden label `metadata_default`. The
actual answer quality is invisible to the current eval.

## Confidence calibration: not what it looks like

ECE of 0.179 looks bad but is benign. `category_confidence` is **bimodal**:
- **0.0** тАФ when no categories are extracted
- **0.90тАУ1.00** тАФ whenever any category is extracted

There are almost no values between 0.1 and 0.85. The 0.7 threshold is effectively a
binary switch. The "miscalibration" (ECE driven by the 0.0тАУ0.1 bin having acc=1.0) just
means: when the router emits no categories, it's usually right to use the hybrid/semantic
path anyway. This is not a problem to fix тАФ it's correct behavior.

## Proposed solutions

### P1 тАФ Fix the golden set labeling (highest impact)

**Problem:** `_derive_router_ground_truth()` assigns labels mechanically from strata.
Questions that escape their stratum framing get wrong labels, making the router look worse
than it is.

**Solution:** After synthesis, run each question through the router and have an LLM
adjudicate: "Given this question, is the router's output or the stratum label more
semantically correct?" Use the LLM verdict as the ground truth.

```python
# In generate_golden_set.py тАФ add a post-hoc re-label step
def adjudicate_label(question, stratum_label, router_result) -> str:
    """Ask an LLM to pick the more correct function label."""
    ...
```

This would likely push true accuracy to ~90% and give a much more reliable eval signal.

### P2 тАФ Force hybrid mode on `*_combined` functions

**Problem:** `metadata_combined` is triggered when `category_confidence >= 0.7` and
`specific_only=False`. But `*_combined` queries have broad framing ("what should I know
about X, especially Y") that benefits from semantic retrieval even when the category
is explicitly named.

**Solution:** In `_derive_retrieval_mode()`, or in `_retrieve_and_rerank()`, override the
mode to `hybrid` whenever `function` ends in `_combined`:

```python
# In router.py _retrieve_and_rerank():
if fn.endswith("_combined") and mode == "metadata":
    mode = "hybrid"
```

Or equivalently in the derivation logic:
```python
def _derive_retrieval_mode(...):
    if not course_ids:
        return "semantic"
    if category_confidence >= CATEGORY_CONFIDENCE_THRESHOLD and specific_only:
        return "metadata"   # only override to metadata for *_specific, not *_combined
    return "hybrid"
```

**Risk:** None on the 34-case TEST_SUITE (no `*_combined` cases have `mode=metadata` in
the current suite). Verify with `eval_pipeline.py --function metadata_combined` before
shipping.

### P3 тАФ Suppress course_id extraction for context-mentioned courses

**Problem:** Case 18 тАФ "If I got a C in Physics 150, does that mean I'm okay to take
**this** class?" тАФ the word "this" implies the user is asking about a course they're
already looking at. "Physics 150" is cited as prerequisite context, not the queried course.
Router extracts `PHYS 150` and tries to retrieve it (it's not in the corpus тЖТ 0 chunks).

**Solution:** Add a prompt rule: "Do not extract a course_id if it is referenced as
prerequisite context for an unnamed course (e.g. 'I got a B in MATH 151, can I take
this course?'). Extract only the course the student is asking about."

### P4 тАФ Add answer quality evaluation (closes the eval gap)

**Problem:** The eval only measures routing correctness. Generated responses are never
scored against `reference_answer`. We have no signal on retrieval recall or answer
faithfulness.

**Two complementary approaches:**

#### 4a тАФ Retrieval recall@k
Check whether the source chunk (identified by `source_crn` + `source_category`) appears
in the retrieved chunks. Already feasible with current data тАФ no new LLM calls needed.

```python
# In eval_pipeline.py run_test():
source_crn = tc.metadata.get("source_crn")
source_cat  = tc.metadata.get("source_category")
hit = any(
    c.get("crn") == source_crn and c.get("category") == source_cat
    for c in chunks
)
recall_hit = hit  # True/False per case
```

This requires passing `source_crn`/`source_category` from the golden set into `TestCase`
(currently stored in `notes` as a workaround).

#### 4b тАФ RAGAS faithfulness + answer relevancy
We already have RAGAS scoring in `db/observability.py` (live traces). The same logic can
be run offline in the eval harness using the `reference_answer` as the ground truth context.

```python
from db.observability import score_ragas
scores = score_ragas(
    question=tc.query,
    answer=response_text,
    contexts=[c["content"] for c in chunks],
    reference=tc.reference_answer,  # needs to be threaded through TestCase
)
```

**Recommended order:** Start with 4a (recall@k) тАФ it's cheap and directly validates whether
the retrieval system is finding the right chunks. Add RAGAS as a secondary signal once
recall is confirmed to be high.

### P5 тАФ Fix the `hybrid_combined` framing issue (minor)

Case 32: "What's the workload like in CSCE 221, **especially considering** the grading?"
was labeled `hybrid_combined` but routed to `hybrid_specific` (`specific_only=True`).
The "especially" phrasing signals `specific_only=False` per the prompt examples, but the
router parsed it as exclusive focus on GRADING.

Update the prompt example to explicitly cover "especially considering" as a `combined`
signal alongside "especially the", "with a focus on", etc.
