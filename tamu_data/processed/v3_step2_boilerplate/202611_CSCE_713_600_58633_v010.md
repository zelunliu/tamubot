[16.0pt] Catalog Description
Software Security. (3-0). Credit 3. Basic principles of design and implementation of
secure software, code reviews including tool-assisted review by static and dynamic
analysis, risk analysis and management, and methods for software security testing.
Prerequisites: CSCE 431 or CSCE 606 or approval of instructor; graduate
classification.

Prerequisite/Corequisite(s):  CSCE 431 or CSCE 606 or approval of instructor;

[16.0pt] Course Learning Outcomes
Upon completion of this course, the learner will be able to:
[11.0pt bold] list  the first principles of security and  explain  why each is important to
security and how it enables the development of security mechanisms that
can implement desired security policies.
[11.0pt bold] identify  specific principles that have been violated in common security
failures.
[11.0pt bold] identify  appropriate design principles to apply in each software development
scenario.
[11.0pt bold] explain  the interaction between security and system usability and
importance of human-computer interfaces to system usability.
[11.0pt bold] explain  the importance of secure software and the programming practices,
development processes, and methodologies that lead to secure software.
[11.0pt bold] explain  techniques for specifying program behavior, the classes of well-
known defects, and how they manifest themselves in various languages.
[11.0pt bold] perform  penetration testing on previously unknown software.
[11.0pt bold] analyze  existing source code for functional correctness.
[11.0pt bold] analyze  software for defects using industry standard tools.
[11.0pt bold] develop  test cases that demonstrate the existence of defects.
[11.0pt bold] develop  defect-free software components that satisfy their functional
requirements.

[11.0pt bold] *Vulnerabilities PoC
Students are required to deliver PoCs for the vulnerabilities explained in class by the
instructor. PoCs should be an end-to-end exploit of the sample applications provided
by the instructor.
The PoC is considered successful if it meets all the requirements below:
The student delivers a source code for the PoC.
The student delivers a video demonstrating the exploitation at the student
machine.
The exploitation can be reproduced by the instructor/TA.
[11.0pt bold] **First Project
Each group should:
Students should identify and exploit an exploitable Web vulnerability in a
Web-based project available on GitHub.
The vulnerability should not have been reported before in any sense (i.e., no
known CVEs)
The students should responsibly disclose the vulnerabilities to the project
creators according to the instructions provided in the lecture on the topic.
Projects without proper disclosure will not be considered.
The exploitation should be done locally (e.g., in Docker) and not in a live
system. Exploitation attempts causing actual harm will not be allowed.
The students should deliver a PoC and a video for the exploitation, as with
the traditional PoC assignments.
The students should deliver a presentation report in class on the final
project dates.
[11.0pt bold] ***Final Project
Each group should:
Students should identify and exploit an exploitable non-Web vulnerability in
a real application,
Web applications are not valid for the final project.
The same rules as for the First project apply.
[11.0pt bold] **** Independent Self-Study
Students are encouraged to deliver a 1-page summary of known vulnerabilities from
the literature, upon the instructor’s suggestion. The evaluation is as follows:
Identifying pre-conditions: 10%
Technical Explanation: 50%
Remediation Explanation: 30%
Contextualization with other class topics: 10%
Topics:
The topics will include vulnerabilities pointed out by the instructor in Canvas
and all final project presentations by all teams.
